# 자유게시판 게시글 기능 요구사항 명세

## 1. 문서 목적

에브리타임 클론 화면설계서를 기반으로 재정의한 요구사항 명세서.

문서 범위
- 게시글 목록 조회
- 게시글 상세 조회
- 게시글 작성
- 댓글 및 대댓글 작성/조회
- 공감/스크랩 처리
- 익명 게시글/댓글 처리
- 정책 검증 및 운영 제재 반영

## 3. 핵심 사용자 시나리오

### 3.1 게시글 탐색

사용자는 자유게시판 목록 화면에 진입하여 게시글 목록을 조회한다. 사용자는 검색어를 입력하여 원하는 게시글을 찾을 수 있고 HOT 게시글 영역을 통해 인기 게시글을 빠르게 탐색할 수 있다.

### 3.2 게시글 상세 확인

사용자는 게시글 목록에서 특정 게시글을 선택하여 상세 화면으로 이동한다. 상세 화면에서는 제목, 본문, 작성자 표시 정보, 작성 시각, 공감 수, 댓글 수, 스크랩 수, 댓글 목록을 확인할 수 있다.

### 3.3 게시글 작성

사용자는 글쓰기 화면에서 제목과 본문을 입력하고 필요 시 이미지를 첨부하고 익명 여부를 선택한 뒤 게시글을 등록한다.

### 3.4 댓글 및 대댓글 작성

사용자는 게시글 상세 화면 하단 입력 바를 통해 댓글을 작성할 수 있다. 특정 댓글을 선택한 상태라면 대댓글을 작성할 수 있어야 한다.

### 3.5 공감/스크랩 반응

사용자는 게시글 상세 화면에서 공감을 누르거나 스크랩할 수 있다. 사용자의 반응 여부는 중복 저장되지 않아야 하며 화면 재진입 시 반영되어야 한다.

## 4. 도메인 모델

### 4.1 Post

게시글의 본문 정보와 상태를 저장하는 Entity

| 필드명 | 타입 예시 | 설명 |
| --- | --- | --- |
| `id` | `Long` | 게시글 ID |
| `board_id` | `Long` or `String` | 게시판 식별자 |
| `author_user_id` | `Long` | 실제 작성자 회원 ID |
| `is_anonymous` | `Boolean` | 익명 게시글 여부 |
| `title` | `String` | 게시글 제목 |
| `content` | `Text` | 게시글 본문 |
| `status` | `Enum` | `PUBLISHED`, `HIDDEN`, `BLOCKED`, `DELETED` |
| `view_count` | `Long` | 조회수 |
| `like_count` | `Long` | 공감 수 |
| `comment_count` | `Long` | 댓글 수 |
| `scrap_count` | `Long` | 스크랩 수 |
| `created_at` | `LocalDateTime` | 생성 시각 |
| `updated_at` | `LocalDateTime` | 수정 시각 |
| `deleted_at` | `LocalDateTime` | 삭제 시각|


### 4.2 Comment

댓글과 대댓글을 동일 엔터티로 관리, 계층 구조는 `parent_comment_id`로 표현한다.

| 필드명 | 타입 예시 | 설명 |
| --- | --- | --- |
| `id` | `Long` | 댓글 ID |
| `post_id` | `Long` | 게시글 ID |
| `author_user_id` | `Long` | 실제 작성자 회원 ID |
| `is_anonymous` | `Boolean` | 익명 댓글 여부 |
| `parent_comment_id` | `Long` nullable | 부모 댓글 ID. null이면 루트 댓글 |
| `content` | `Text` | 댓글 본문 |
| `status` | `Enum` | `PUBLISHED`, `HIDDEN`, `BLOCKED`, `DELETED` |
| `created_at` | `LocalDateTime` | 생성 시각 |
| `updated_at` | `LocalDateTime` | 수정 시각 |
| `deleted_at` | `LocalDateTime` | 삭제 시각, 선택값 |

### 4.4 PostReaction

게시글 - 사용자 반응 N:N 중간 테이블.

| 필드명 | 타입 예시 | 설명 |
| --- | --- | --- |
| `id` | `Long` | 반응 ID |
| `post_id` | `Long` | 게시글 ID |
| `user_id` | `Long` | 사용자 ID |
| `type` | `Enum` | `LIKE`, `SCRAP` |
| `created_at` | `LocalDateTime` | 생성 시각 |

제약 조건
- `UNIQUE (post_id, user_id, type)`

### 4.5 Report

신고 데이터를 저장하여 관리자 검토 및 자동 숨김 정책에 활용한다.

| 필드명 | 타입 예시 | 설명 |
| --- | --- | --- |
| `id` | `Long` | 신고 ID |
| `target_type` | `Enum` | `POST`, `COMMENT` |
| `target_id` | `Long` | 대상 ID |
| `reporter_user_id` | `Long` | 신고자 ID |
| `reason_code` | `Enum` | 신고 사유 코드 |
| `description` | `String` | 상세 설명, 선택값 |
| `created_at` | `LocalDateTime` | 신고 시각 |

### 4.6 ModerationLog

자동 검수 및 운영자 조치 이력을 저장

| 필드명 | 타입 예시 | 설명 |
| --- | --- | --- |
| `id` | `Long` | 이력 ID |
| `target_type` | `Enum` | `POST`, `COMMENT` |
| `target_id` | `Long` | 대상 ID |
| `action_type` | `Enum` | `AUTO_HIDE`, `BLOCK`, `RESTORE`, `DELETE` 등 |
| `reason` | `String` | 조치 사유 |
| `actor_type` | `Enum` | `SYSTEM`, `ADMIN` |
| `created_at` | `LocalDateTime` | 처리 시각 |

## 5. 상태 정의

### 5.1 게시글 상태

| 상태 | 설명 |
| --- | --- |
| `PUBLISHED` | 정상 게시 상태 |
| `HIDDEN` | 일반 사용자에게 비노출 |
| `BLOCKED` | 정책 위반으로 차단 |
| `DELETED` | 삭제 처리됨 |

### 5.2 댓글 상태

| 상태 | 설명           |
| --- |--------------|
| `PUBLISHED` | 정상 노출        |
| `HIDDEN` | 일반 사용자에게 비노출 |
| `BLOCKED` | 정책 위반으로 차단   |
| `DELETED` | 삭제 처리됨       |


### 5.3 조치 이력
| 상태 | 설명                    |
| --- |-----------------------|
| `AUTO_HIDE` | 시스템이 신고 누적, 금칙어 탐지 등 정책에 따라 자동 숨김 처리 |
| `BLOCK` | 관리자 또는 시스템이 정책 위반으로 차단|
| `RESTORE` | 숨김/차단/삭제된 게시글이나 댓글을 다시 복구됨|
| `DELETE` | 운영자 또는 사용자 요청에 의해 삭제 처리|

## 6. API 요구사항 (초안, SpringBoot 마이그레이션 시 구체화 예정)

### 6.1 게시글 목록 조회

- `GET /boards/free/posts`

#### 처리 규칙

- `PUBLISHED` 상태의 게시글만 기본 조회 대상이다.
- `HIDDEN`, `BLOCKED`, `DELETED` 게시글은 일반 사용자 목록에서 제외한다.
- 검색어가 존재하면 제목, 본문 기준 검색을 수행한다.
- 인기 게시글 조건은 별도 정책으로 정의한다.

### 6.2 게시글 상세 조회

- `GET /posts/{postId}`

#### 처리 규칙

- 존재하지 않는 게시글은 `404`를 반환한다.
- 삭제/차단 상태는 일반 사용자에게 `404` 또는 권한 없는 접근으로 처리한다.
- 조회수 증가는 중복 집계 방지 정책이 필요하다.

### 6.3 게시글 작성

- `POST /posts`

#### 처리 규칙

- 인증된 사용자만 요청 가능하다.
- 정책 검수 결과에 따라 즉시 게시되지 않고 `HIDDEN` 또는 `BLOCKED` 상태로 저장될 수 있다.

### 6.4 게시글 수정

- `PATCH /posts/{postId}`
- 
#### 처리 규칙

- 작성자 본인 또는 관리자만 가능하다.
- 삭제/차단 상태의 게시글은 수정 불가하다.
- 수정 후에도 정책 검수를 재실행이 필요하다.

### 6.5 게시글 삭제

- `DELETE /posts/{postId}`

#### 처리 규칙

- 작성자 본인 또는 관리자만 가능하다.
- 소프트 삭제

### 6.6 게시글 공감 등록

- `POST /posts/{postId}/like`

#### 처리 규칙

- 인증된 사용자만 가능하다.
- 동일 게시글에 대해 사용자당 1회만 허용한다.
- 이미 공감한 경우 멱등 처리 또는 `409`를 반환.

### 6.7 게시글 공감 취소

- `DELETE /posts/{postId}/like`

### 6.8 게시글 스크랩 등록

- `POST /posts/{postId}/scrap`

### 6.9 게시글 스크랩 취소

- `DELETE /posts/{postId}/scrap`

### 6.10 댓글 목록 조회

- `GET /posts/{postId}/comments`

#### 처리 규칙

- 삭제된 댓글은 정책에 따라 본문을 대체 문자열로 반환할 수 있다.
- 현재 요구사항에서는 대댓글 깊이를 1단계로 제한.

### 6.11 댓글 작성

- `POST /posts/{postId}/comments`

#### 처리 규칙

- `parentCommentId`가 null이면 댓글, 값이 있으면 대댓글이다.
- 부모 댓글이 존재하지 않거나 같은 게시글 소속이 아니면 실패해야 한다.

### 6.12 댓글 수정

- `PATCH /comments/{commentId}`

### 6.13 댓글 삭제

- `DELETE /comments/{commentId}`

### 6.14 신고

- `POST /reports`

## 7. 요청/응답 필드 설계 원칙

### 7.1 작성자 정보 노출 원칙

- 내부 저장은 항상 실제 `author_user_id` 기준으로 한다.
- 응답에는 화면 표시용 작성자 정보만 내려준다.
- 익명 게시글/댓글인 경우 실제 회원 식별 정보는 외부에 노출하지 않는다.

### 7.2 카운트 필드 운영 원칙

- `like_count`, `comment_count`, `scrap_count`, `view_count`는 성능을 위해 게시글 테이블에 중복 저장 가능하다.
- 단, 정합성 보정을 위한 배치 또는 이벤트 기반 보정 전략이 필요하다.

### 7.3 페이지네이션 원칙

- 목록과 댓글은 커서 기반 페이지네이션 적용 가능.
- 정렬 기준과 커서 구조는 함께 설계되어야 한다.

## 8. 검증 규칙

### 8.1 게시글 작성 검증

- 로그인 사용자만 작성 가능
- 제목 필수
- 본문 필수
- 제목 공백-only 입력 불가
- 본문 공백-only 입력 불가
- 제목 최대 길이 제한 필요
- 본문 최대 길이 제한 필요
- 허용되지 않은 HTML 또는 스크립트 입력 차단
- 금칙어, 욕설, 비방, 개인정보 포함 여부 검수
- 도배 방지를 위한 작성 빈도 제한

예시:

- 제목 최대 100자
- 본문 최대 5,000자
- 이미지 최대 10개
- 이미지 1개당 최대 10MB

### 8.2 게시글 수정 검증

- 작성자 본인 또는 관리자만 가능
- 삭제된 게시글 수정 불가
- 차단된 게시글 수정 불가 또는 관리자만 허용

### 8.3 목록 조회 검증

- `size` 최대값 제한
- 검색어 길이 제한
- 허용되지 않은 정렬 값 차단
- 잘못된 커서 값 차단

### 8.4 상세 조회 검증

- 존재하지 않는 게시글 접근 차단
- 삭제/숨김/차단 게시글 접근 정책 분리
- 권한 없는 사용자의 비공개 리소스 접근 차단

### 8.5 댓글 작성 검증

- 로그인 사용자만 작성 가능
- 본문 필수
- 공백-only 입력 불가
- 댓글 최대 길이 제한 필요
- 부모 댓글 존재 여부 검증
- 부모 댓글과 게시글 ID 일치 여부 검증
- 대댓글의 재대댓글 작성 차단
- 삭제/차단된 게시글에는 댓글 작성 불가

권장 예시:

- 댓글 최대 1,000자

### 8.6 공감/스크랩 검증

- 로그인 사용자만 가능
- 게시글 존재 여부 확인
- 삭제/차단 게시글에 대한 반응 차단
- 중복 반응 차단

### 8.7 익명 처리 검증

- 익명 허용 게시판인지 확인
- 익명 작성이어도 내부적으로 작성자 식별 가능해야 함
- 응답 데이터에서 실제 회원 식별값 노출 금지

### 8.8 정책 검수

아래 항목은 자동 검수 또는 관리자 검토 대상으로 본다.

- 욕설 및 혐오 표현
- 특정 개인에 대한 비방
- 연락처, 계좌번호, 주소 등 개인정보
- 성적 표현 또는 불법 정보
- 반복 게시 또는 도배

## 9. 비기능 요구사항

### 9.1 성능

- 게시글 목록 조회는 페이지 단위 응답 시간이 안정적으로 유지되어야 한다.
- 게시글 상세 조회 시 댓글 개수가 많아도 과도한 지연이 없어야 한다.
- 카운트 집계와 반응 저장은 동시성 충돌에 대한 고려가 필요하다.

### 9.2 정합성

- 공감/스크랩/댓글 수는 원본 데이터와 집계 데이터 간 불일치가 장기화되지 않아야 한다.
- 댓글 생성/삭제 시 `comment_count` 보정 전략이 필요함.

### 9.3 보안

- 인증되지 않은 사용자 요청 차단
- 익명 작성자의 실제 식별 정보 비노출
- XSS 및 악성 스크립트 입력 방지
- 파일 업로드 시 악성 파일 검증

### 9.4 운영성

- 신고 누적 기준 자동 숨김 정책을 정의할 수 있어야 한다.
- 관리자 검토 이력이 남아야 한다.
- 삭제/차단 이력 추적이 가능해야 한다.

## 10. 에러 응답 기준

초기엔 기본 응답 기준만 정의함

| 상황 | 권장 상태 코드 | 설명 |
| --- | --- | --- |
| 인증 필요 | `401` | 로그인 필요 |
